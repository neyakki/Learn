## Имена

Имена должны передавать намерения программистов. Следите за именами в программе и изменяйте их, если найдется более
удачное имя. Если имя требует дополнительных комментариев, то оно не передает всех намерений программиста. Правильно
выбранные имена дополняют структуру кода. Имена должны описывать все побочные эффекты.

Например:

```python
# Плохо
d = "12:43"

# Лучше
fileAgeInDays = "12:43"
```

Проблема кроется в _неочевидности_ кода, т.е. из контекста кода нельзя понять его смысл.

Не используйте слова со скрытыми значениями, отличными от предполагаемого. Например: `os, pid`.

Остерегайтесь малозаметных различий:

-   `XYZControlEfficientHandling`;
-   `XYZControlEfficientScroll`

Имена должны обозначать разные понятия, если они различаются.

Например:

```python
# Плохо
def test(a1, a2): ...


# Лучше
def copy(source, destination): ...
```

Используйте удобно произносимые имена:

Например:

```python
# Плохо
genytms = ...

# Лучше
generate_timestamp = ...
```

Не загромождайте имена венгерской записью. Инфомрация о типах или области видимости не должна кодироваться в именах.

Используйте имена удобные для поиска. Однобуквенные имена и числовые константы трудно искать в большой кодовой базе.

Длинные имена лучше коротких, а имена, удобные для поиска, лучше, чем константы в коде

Длинные имена должны соответствовать размеру его области видимости

Имя **классов** и **объектов** должно состоять из существительных и их комбинаций.

```python
# Плохо
class CarDrive(): ...


# Лучше
class Car(): ...
```

Имя **методов** и **функций** должно состоять из глаголов или глагольных словосочетаний. Чем меньше и специализированнее
функция, тем проще выбрать для нее имя. Поиск хорошего имени для функции может привести к полезной реструктуризации
кода.

```python
# Плохо
def name(): ...


# Лучше
def get_name(): ...
```

В имени метода следует отобразить все, что выполняет функция, если имя слишком длинное, то значит функция требует разбивки на более специфичные. Возвращающийся результат, также стоит описывать в имени функции.

Выберите одно слово для преставления одной абстрактной концепции.

Используйте термины из области информатики: названия алгоритмов, паттернов и т.д.

Используйте имена из предметной области.

Имена следует помещать в определенный контекст, например, в классы, функции и т.д.

## Классы

Класс - набор данных и методов, имеющих общую, хорошо определенную зону ответственности.

Класс должен начинаться со списка переменных:

-   Статические константы;
-   Статические переменные;
-   Приватные переменные

Затем идет список открытых функций. Приватные функции, вызываемые открытыми функциями, размещаются непосредственно за самой открытой функцией.

Классы должны быть компактными. Размер класса определяется его ответственностью. Имя класса описывает его ответственность. Если для класса не удается подобрать четкое имя, то значит классы слишком велик.

Система должна состоять из множества маленьких классов, где каждый класс инкапсулирует одну ответственность, имеет одну причину для изменения и взаимодействует с другими классами.

Класс, в котором каждая переменная используется в каждом методе, обладает максимальной связанностью. Это означает, что методы и переменные существуют как единое целое.

Если клиентский класс зависит от конкретных подробностей, то их изменение может привести к нарушению его работоспособности. Чтобы изолировать воздействие этих подробностей на класс вводят интерфейсы и абстрактные классы.

Интерфейс должен представлять хорошую абстракцию, скрывающую детали реализации класса. Абстракция понимается представление сложной операции в упрощенной форме. Интерфейс класса - абстракция реализации класса, скрытой за интерфейсом. Интерфейс класса предоставляет четко согласующуюся группу методов.

Интерфейс класса должен выражать согласованный уровень абстракции.

!!! Note "Закон Деметры"

    Модуль не должен знать внутренне устройство тех объектов, с которыми он работает. Это значит, что объект не должен раскрывать свою внутреннюю структуру через методы доступа.

    В точной формулировке закон Деметры гласит, что метод *f* класса *C* должен ограничиться вызовом следующих объектов:

    * C;
    * Объекты созданные f;
    * Объекты, переданные в аргументы f;
    * Объекты, хранящиеся в переменной экземпляра C.

    Метод не должен вызывать методы объектов, возвращаемых любыми из разрешенных методов.

    !!! Failure "Так нельзя"

        account.ContactPerson().DaytimeContactInfo()

Объекты скрывают свои данные за абстракциями и предоставляет функции, работающие с этими данными.

Структуры данных раскрывают свои данные и не имеют осмысленных функций.

Процедурный код (структуры данных) позволяют легко добавлять новые функции без измерения существующих структур данных.

ОО код упрощает добавление новых классов без изменения существующих функций.

Инкапсуляция не позволяет узнать детали реализации. Если не удается понять каким делать метод: открытый, закрытый, защищенный, то стоит выбираться самый строгий уровень. Предоставление доступа к данным-членам должен выполняться через специализированные методы. Интерфейс не должен содержать закрытые детали реализации.

При наследовании класса:

-   новый класс является специализированной версией существующего класса;
-   Убедитесь, что происходит наследование только используемого функционала;
-   Перемещайте общий функционал на более высокий уровень иерархии абстракции;

Когда использовать наследование и композицию:

-   Если классы имеют общие данные, но не поведение, создайте общий класс, в который можно включить во все классы;
-   Если несколько классы имеют общее поведение, но не данные, сделайте эти классы производные от общего класса;
-   Если несколько классов имеют общее поведение и данные, сделайте эти классы производными от общего;
-   Используйте наследование, если хотите, чтобы интерфейс определялся базовы классом, и включение, если хотите сами контролировать интерфейс.

## Функции и методы

Функции должны быть компактными.

Функции должны выполнять только одну операцию и ничего другого. Если функция выполняет только действия, находящиеся на
одном уровне абстракции, то функция выполняет одну операцию.

Код должен читаться сверху вниз как рассказ. За каждой функцией должны идти функции следующего уровня абстракции.

Аргументы функции (от лучшего к худшему):

-   Функции не должна принимать никаких аргументов (нульарная);
-   Функция может принимать один аргумент (унарная);
-   Функция может принимать два аргумента (бинарная);
-   Функция может принимать три аргумента (тернарная)

Если функция должна получить более двух аргументов, то скорее всего некоторые из аргументов можно упаковать в класс.

Передача логического значения функции (аргумент-селектор) усложняет ее, т.е. функция может выполнять более одной
операции. Лучше иметь несколько функций.

Функция должна делать только то, что она обещает и ничего больше.

Если функция должна изменять чье-то состояние, то это должно быть состояние своего владельца.

Функция должна что-то делать или отвечать на вопрос, но не одновременно.

```python
# Плохо
def set(attr, value) -> bool: ...


# Лучше
def is_exists(attr) -> bool: ...


def set_attr(attr, value) -> None: ...
```

Обработка ошибок - одна операция, поэтому ее следует выделять в отдельную функцию.

Если функция остается компактной, то добавление `return`, `break`, `continue` не принесет вреда, а возможно сделает код
более выразительным.

Временные привязки не должны скрываться. Каждая функция передает результат следующей, тем самым не получится нарушить
порядок выполнения.

```python
# Плохо
saturate_gradient()
retriculate_splines()
dive_for_moog()


# Лучше
gradient = saturate_gradient()
splines = retriculate_splines(gradient)
dive_for_moog(splines)
```

Важная причина создания методов - снижение сложности программы. Метод создается для сокрытия информации, чтобы о ней не думать. Одним из признаков разделения методов является глубокая вложенность.

Также создание методов позволяет сократить дублирование кода и снизить уровень ошибок.

Функция позволяет упростить проверку сложных булевских проверок: проверка будет иметь осмысленное имя

Функции позволяют выполнить оптимизацию в одном месте.

!!! Note

    Создавать методы для простых операций не плохо.

Связанность функций (от лучшей к худшей):

* Функциональная - метод выполняет только одну операцию. Методы соответствуют своим именам;
* Последовательная - методы выполняются в строго определенном порядке, используя данные предыдущих этапов;
* Коммуникационная - методы используют общие данные, но не связанны между собой;
* Временная - методы объединены, т.к. выполняются в один момент времени;
* Процедурная - методы выполняются в определенном порядке;
* Логическая - выполнение операции зависит от управляющего флага. Считается удачным если роль метода состоит только в координации выполнения команд (Обработчик событий);
* Случайная - между операциями нет какой либо связи

Все передаваемые аргументы должны использоваться при выполнении метода.

Не стоит использовать передаваемые параметры как рабочие переменные.

Функция должна принимать не более 7 аргументов.

Передавать в метод нужна те переменные, которые нужны для поддержания его абстракции интерфейса.

Если абстракция ожидает три конкретных элемента, которые принадлежат одному объекту, то и передаем их по отдельности. Если данные принадлежат конкретному объекту, над которым метод выполняет операцию, то передавая элементы по отдельности, мы нарушаем абстракцию.

Используйте именованные параметры.

## Комментарии

Бессодержательные комментарии загромождают код.

Чем древнее комментарий, чем дальше он от кода, который описывает, тем больше он лжет. Только код может правдиво сообщить, что он делает.

Ясный и выразительный код с минимумом комментариев лучше сложного и громоздко кода с большим количеством комментариев.

Чтобы объяснить намерения кода, достаточно создать функцию, которая будет сообщать тоже самое что и комментарий.

!!! Note

    Хороший комментарий без которого можно обойтись.

**Хорошие комментарии:**

-   Комментарии **TODO** являются полезными, т.к. они позволяют оставить заметку для реализации необходимого функционала, но сейчас это сделать не возможно;
-   Комментарии, предупреждающие о нежелательных последствиях;
-   Иногда полезно включить в комментарий пояснение к коду, но если возможно, информацию лучше передавать в имени функции;
-   Комментарии могут выходить за рамки полезной информации и описывать намерения, заложенные разработчиком;
-   Комментарии, подчеркивающие важность кода.

!!! Note

    Плохие комментарии представляют собой "подпорки" для плохого кода или оправдания сомнительных решений.

**Плохие комментарии:**

-   Не стоит добавлять комментарии только потому, что кажется, что это уместно;
-   Лишние заголовочные комментарии. Такие комментарии читается не проще, чем сам код;
-   Комментарии, неточные и не соответствующие истине;
-   Комментарии, утверждающие очевидное;
-   Закомментированный код.

## Форматирование

Имя файла должно быть простым, но содержательным. Его достаточно, чтобы читатель понял и открыл нужный модуль. Начало файла должно описывать высокоуровневые концепции и алгоритмы. Степень детализации увеличивается к концу файла.

Мелкие файлы более понятны, чем большие.

Каждая строка представляет выражение или условие, а каждая группа строк представляет законченную мысль. Мысли отделяются друг от друга пустыми строками.

Концепции, связанные друг с другом, должны находиться рядом.

Переменные следует объявлять как можно ближе к месту использования.

Пример:

```python
run = True
while run:
    ...
```

Переменные экземпляра, должны объявляться в начале класса.

Пример:

```python
class Test:
    def __init__(self):
        self.test = test
```

Зависимые функции должны располагаться рядом друг с другом, вызывающая функции должна находиться над вызываемой.

Идеальный размер горизонтального форматирование - 120 символов.

## Обработка ошибок

Обработка ошибок важна, но если она заслоняет собой логику программы, то она реализована не правильно.

Создавайте содержательное сообщение об ошибках и передавайте их со своими исключениями.

Ошибки можно классифицировать по разным признакам: по источнику, по компоненту, сбои устройств и т.д.

Если происходит перехват множества исключений и выполняется однотипное действие не зависимо от исключения, то можно создать обертку, возвращающая стандартное исключение:

```python
class CustomError(Exception):
    pass

def wrapper():
    try:
        ...
    except Exception1:
        raise CustomError()
    except Exception2:
        raise CustomError()
    except Exception3:
        raise CustomError()
    except Exception4:
        raise CustomError()

def main():
    try:
        wrapper()
    except CustomError:
        print("error")
```

Инкапсулируя перехват ошибок в функцию, мы сокращаем зависимость от стороннего вызова.

Паттерн "Особый случай" поваляет настраивать объект так, чтобы он обрабатывал особый случай, без использования обработки исключительного поведения в клиентском коде.

## Тесты

По требованиям TDD модульные тесты должны писаться заранее, еще до написания кода проекта.

Законы:

1. Не пишите код продукта, пока не напишите отказной модульный тест;
2. Не пишите модульный тест в объеме большем, чем необходимо для отказа;
3. Не пишите код продукта в объеме большем, чем необходимо для прохождения текущего отказного теста.

!!! Note

    Тестовый код не менее важен, чем код продукта

Модульные тесты обеспечивают гибкость, удобство и возможно повторного использования кода.

Если у нас имеются тесты, то мы не боимся вносить изменения в код.

В тестовом коде необходимо передавать максимум информации минимумом выразительных средств.

Каждый тест должен содержать только одну проверку _assert_ и тестировать только одну концепцию.

Чистые тесты должны обладать следующими характеристиками:

-   Тесты должны выполняться быстро;
-   Тесты не должны зависеть друг от друга;
-   Тесты должны давать повторяемые результаты;
-   Результатом выполнения теста должен быть логический признак;
-   Тесты должны создаваться своевременно.
