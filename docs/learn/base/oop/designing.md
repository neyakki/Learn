# Проектирование классов

ООП поддерживает идею создание классов в виде полных пакетов, которые инкапсулируют данные и поведения единого целого. Класс должен представлять логический компонент.

## Моделирование реальных систем

Одна из основных целей ООП - моделирование реальных систем способами, схожими с фактическим образом мышления людей. Проектирование классов - это объектно-ориентированный вариант создания таких моделей. Плюсы этого подхода в том, что классы буквально моделирует реальные объекты и их взаимодействие с другими такими же объектами. Поэтому при создании классов их следует проектировать их тем способом, который позволит представить истинное поведение объекта.

## Определение открытых интерфейсов

Наиболее важная задача при проектирование класса - определение минимального открытого интерфейса. Интерфейс хорошо спроектированного объекта описывает услуги, оказание которых требуется клиенту. Если класс не будет предоставлять полезных услуг пользователям, его не стоит создавать.

### Минимальный открытый интерфейс

Обеспечение минимального открытого интерфейса позволяет сделать класс более компактным. Цель состоит в том, чтобы пользователям тот интерфейс, который действительно им нужен для правильной работы. Если для открытого интерфейса не будет предусмотрено соответствующих ограничений, то могут возникнуть проблемы с целостностью и защитой системы.

!!! note "Расширение интерфейса"

    Если открытого интерфейса будет не достаточно, то с помощью объектной технологии можно с легкостью его расширить и адаптировать.

!!! info "Пример"

    Допустим у нас имеется два класса **Сотрудник** и **Бухгалтер**. Если классу **Сотрудник** потребуется предоставить другим объектам значение атрибута *name*, то класс **Сотрудник** должен определить открытый метод *getName()*. Теперь класс **Бухгалтер** сможет получит значение *name*, то ему придется всего лишь вызвать метод *getName()*.

Пользователям не нужно знать о внутренней работе класса, им важно знать только как создавать и использовать объекты.

### Сокрытие реализации

Определение открытого интерфейса - задача проектирования, вращающаяся вокруг пользователей класса, то реализация не должна их касаться. Изменения в реализации не должны повлечь за собой изменения в пользовательском коде. Лучший способ избежать этого, использовать интерфейсы и композицию.

## Проектирование надежных конструкторов (деструкторов)

Прежде всего конструктор должен задавать для объекта его начальное, надежное состояние.

При использовании ЯП, в которых имеются деструкторы, тогда эти деструкторы должны содержать функции по отчистке. Код должен содержать надежное высвобождение памяти, которую объект занимал во время своего существования. Если проигнорировать это, то может произойти утечка памяти.

## Проектирование с учетом повторного использования

Объекты могут повторно использоваться в разных системах, а код следует писать с учетом такого повторного использования. Чтобы сделать класс пригодным к использованию в разных системах, его нужно проектировать с учетом повторного использования.

## Проектирование с учетом расширяемости

Добавление новых функций в класс может быть таким же простым, как расширение существующего класса, добавление нескольких новых методов и модификаций поведений других. Именно здесь в роль вступает наследование.

!!! info "Пример"

    Например, у нас имеется класс **Person**, который будет являться суперклассом для классов **Employee**. В этом случае класс **Person** является расширяемым. Поэтому проектировать класс **Person** таким способом, чтобы он не мешал расширяемости дочерними классами.

Этот аспект затрагивает рекомендации по абстрагированию. Класс **Person** должен содержать только данные и поведение, специфичные для него.

## Проектирование с учетом сопровождаемости

Проектирование практичных и компактных классов обеспечивает высокий уровень сопровождаемости. Процесс проектирования классов вынуждает нас разделять код на множество управляемых фрагментов. Отдельные фрагменты удобнее в сопровождении, чем их большие фрагменты. Один из лучших способов обеспечить сопровождаемость - уменьшение количества взаимосвязанного кода. Т.е. изменения в одном классе не должны сказываться, даже минимально, на других.

Если классы изначально правильно спроектированы, то любые изменения в системе должны вноситься только в реализацию объекта, а изменения интерфейса должны избегаться любой ценной. Любые изменения интерфейса имеют волновой эффект для всей системы.
